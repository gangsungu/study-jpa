## 10장. 객체지향 쿼리 언어

### JPQL이란?
+ JPA 표준에서 제공하는 객체지향 쿼리 언어
+ Entity 객체 기준으로 데이터를 질의
+ SQL처럼 보이지만 DB 테이블/컬럼이 아닌 엔티티/필드를 사용한다
+ JPA가 JPQL을 해석하여 `실제 SQL로 변환` 후 DB에 보낸다

### JPQL 기본문법
+ SELECT 문법
    ```java
    String jpql = "SELECT m FROM Member m WHERE m.age > 20";
    List<Member> members = em.createQuery(jpql, Member.class)
                            .getResultList();

    // Member : 엔티티 클래스명
    // m : 엔티티의 별칭
    // m.age : 엔티티의 필드명
    ```

+ 파라미터 바인딩
    - 이름 기반 바인딩
        ```java
        String jpql = "SELECT m FROM Member m WHERE m.name = :name";
        List<Member> result = em.createQuery(jpql, Member.class)
                            .setParameter("name", "강선구")
                            .getResultList();
        ```

    - 위치 기반 바인딩
        ```java
        String jpql = "SELECT m FROM Member m WHERE m.name = ?1 AND m.age = ?2";
        List<Member> result = em.createQuery(jpql, Member.class)
            .setParameter(1, "강선구")
            .setParameter(2, 30)
            .getResultList();
        ```

+ 프로젝션(projection)
    - 엔티티 프로젝션
        + 엔티티 객체를 반환
        ```java
        String jpql = "SELECT m FROM Member m";
        List<Member> result = em.createQuery(jpql, Member.class).getResultList();

        // Member 엔티티를 반환
        ```

    - 스칼라(Scalar) 프로젝션
        ```java
        String jpql = "SELECT m.name FROM Member m";
        List<String> names = em.createQuery(jpql, String.class).getResultList();

        String jpql = "SELECT m.name, m.age FROM Member m";
        List<Object[]> result = em.createQuery(jpql).getResultList();
        for (Object[] row : result) {
            String name = (String) row[0];
            Integer age = (Integer) row[1];
        }
        ```

    - DTO 프로젝션 : DTO 객체로 매핑해서 반환
        ```java
        String jpql = "SELECT new com.example.dto.MemberDto(m.name, m.age) FROM Member m";
        List<MemberDto> dtos = em.createQuery(jpql, MemberDto.class).getResultList();

        // DTO 객체 안에 파라미터 순서 타입과 맞는 생성자가 필요
        ```

    - 임베디드 프로젝션
        ```java
        String jpql = "SELECT m.address FROM Member m";
        List<Address> addresses = em.createQuery(jpql, Address.class).getResultList();

        // 임베디드 타입을 반환한다
        ```

        | 프로젝션 종류 | 특징 | 장점 | 단점/주의사항 |
        | --- | --- | --- | --- |
        | 엔티티 | 엔티티 객체 그대로 반환 | 관리 편함, 엔티티 직접 사용 | Lazy 로딩, N+1 문제, 불필요한 데이터 포함 가능 |
        | 단일/다중 필드 | 일부 값만 추출 (String, Object\[], etc.) | 가볍고 빠름 | 여러 컬럼이면 Object\[] → 타입 안전성 낮음 |
        | DTO(new) | DTO로 직접 매핑 | 타입 안전, 필요한 데이터만 전달 | 생성자 필요, 필드명·타입 순서 주의 |
        | 임베디드 타입 | 임베디드(내장) 타입 반환 | 복합 값 객체 직접 반환 가능 | - |

### 타입 쿼리와 일반 쿼리
+ TypedQuery
    - 제네릭 기반으로 반환 타입을 명확하게 지정
    - createQuery(jpql, 반환타입.class)
    - 반환값이 엔티티, 단일속성, DTO 등 명확할 때 사용
        ```java
        String jpql = "SELECT m FROM Member m WHERE m.name = :name";
        TypedQuery<Member> query = em.createQuery(jpql, Member.class);
        query.setParameter("name", "강선구");
        List<Member> result = query.getResultList();
        ```

+ Query
    - 반환 타입이 미지정
    - 반환 타입이 여러개(여러 필드, 여러 타입, Object[] 등)
        ```java
        String jpql = "SELECT m.name, m.age FROM Member m";
        Query query = em.createQuery(jpql);
        List<Object[]> result = query.getResultList();
        for (Object[] row : result) {
            String name = (String) row[0];
            Integer age = (Integer) row[1];
        }
        ```

        | 구분 | TypedQuery<T> | Query |
        | --- | --- | --- |
        | 타입 | 반환 타입 명확 (제네릭 지정) | 반환 타입 미정 (Object 등) |
        | 안전성 | 컴파일 타임에 타입 체크 | 런타임에 캐스팅 필요 |
        | 반환 예시 | Member, String, DTO 등 | Object\[], Object |
        | 사용 권장 | 가능하면 TypedQuery 사용 | 복합 결과 등 불가피할 때 |

### 페이징
+ setFirstResult(int startPosition)
    - 0부터 시작, (pageNumber - 1) * pageSize
+ setMaxResults(int maxResult)
    - 한번에 가져올 데이터의 개수(페이지 크기)
        ```java
        String jpql = "SELECT m FROM Member m ORDER BY m.id DESC";
        List<Member> result = em.createQuery(jpql, Member.class)
            .setFirstResult(0)  // 시작 위치(0부터)
            .setMaxResults(10)  // 가져올 개수(페이지 크기)
            .getResultList();
        ```

### 집합 함수와 그룹화
+ 집합 함수 : COUNT, SUM, AVG, MAX, MIN
    - `COUNT`는 데이터가 없으면 0을 반환
    - `SUM, AVG, MAX, MIN`은 데이터가 없으면 null을 반환 (주의)
+ GROUP BY : 특정 그룹끼리 묶어서 통계
+ HAVING : 그룹화한 통계 결과에 대한 조건 (WHERE는 그룹화 이전, HAVING은 그룹화 이후에 필터링)
    ```java
    SELECT t.name, COUNT(m), AVG(m.age)
    FROM Member m JOIN m.team t
    GROUP BY t.name
    HAVING AVG(m.age) >= 30
    ```

### 정렬 (ORDER BY)
+ ASC(오름차순, 기본값), DESC(내림차순)
    ```java
    SELECT m FROM Member m ORDER BY m.age DESC, m.name ASC
    ```

### 경로 표현식
+ `.`(점)을 찍어 객체 그래프를 탐색하는 것
    - 상태 필드(state field) : 단순히 값을 저장하는 필드 (`m.name`, `m.age`)
        + 경로 탐색의 끝, 더 이상 탐색 불가
    - 단일 값 연관 필드(`@ManyToOne`, `@OneToOne`) : `m.team`
        + 묵시적 내부 조인 발생, 계속 탐색 가능 (`m.team.name`)
    - 컬렉션 값 연관 필드(`@OneToMany`, `@ManyToMany`) : `t.members`
        + 묵시적 내부 조인 발생, 더 이상 탐색 불가
        + 단, FROM 절에서 별칭을 얻으면 그 별칭으로 다시 탐색 가능
+ 묵시적 조인은 SQL상 내부 조인만 가능하고 쿼리 튜닝이 어려우므로 `명시적 조인`(JOIN) 사용을 권장

### 조인
+ 관계형 데이터베이스에서는 두 테이블을 외래 키 등을 사용하여 논리적으로 연결해 원하는 데이터를 조회하는 것
+ JPA에서는 엔티티 간의 연관관계를 바탕으로 JPQL, QueryDSL 등을 사용하여 데이터를 조회하는 것
    - 내부 조인 (inner join)
        + 조건에 맞는 양쪽 엔티티의 데이터가 모두 존재할 때, 결과에 포함
        + SQL의 inner join과 똑같다
            ```java
            SELECT m FROM Member m JOIN m.team t WHERE t.name = '팀A'
            ```
            - Member와 Team이 연관관계라면 member에 연관된 team이 있을때만 조회됨

    - 외부 조인(Outer join)
        + 왼쪽 엔티티는 항상 가져오고 연관 엔티티가 없으면 null
            ```java
            SELECT m FROM Member m LEFT JOIN m.team t
            ```

    - 세타 조인(Theta join)
        + 연관관계가 없는 엔티티끼리 조인할 때 사용
            ```java
            SELECT m, t FROM Member m, Team t WHERE m.name = t.name
            ```

    - on 절
        + 연관관계가 없는 엔티티끼리도 조인가능
        + 연관관계가 있어도 원하는 조건을 추가할 수 있음
            ```java
            SELECT m, t
            FROM Member m
            LEFT JOIN Team t ON m.name = t.name
            ```
            - Member와 Team이 연관관계가 없어도 ON 절로 조인이 가능

            ```java
            SELECT m, t
            FROM Member m
            LEFT JOIN m.team t ON t.name = 'A팀'
            ```
            - m.team은 연관관계지만 추가 조건으로 'A팀'인 경우만 조인에 포함시킴

### 페치 조인 (fetch join)
+ JPQL에서 `성능 최적화`를 위해 제공하는 기능
+ 연관된 엔티티나 컬렉션을 SQL 한 번에 함께 조회 → `N+1 문제 해결`
+ `JOIN FETCH` 명령어 사용 (SQL의 조인 종류가 아니라 JPQL 전용 문법)
    - 엔티티 페치 조인
        ```java
        SELECT m FROM Member m JOIN FETCH m.team
        ```
        + 지연 로딩(LAZY)으로 설정했어도 페치 조인이 우선
        + 프록시가 아닌 `실제 엔티티`가 함께 조회되어 채워짐

    - 컬렉션 페치 조인
        ```java
        SELECT t FROM Team t JOIN FETCH t.members WHERE t.name = '팀A'
        ```
        + 일대다 조인이므로 결과가 `뻥튀기(중복)`될 수 있음
        + `DISTINCT`로 중복 제거 (JPQL의 DISTINCT는 SQL DISTINCT + 애플리케이션에서 엔티티 중복도 제거)
        ```java
        SELECT DISTINCT t FROM Team t JOIN FETCH t.members
        ```

+ 페치 조인의 한계
    - 페치 조인 대상에는 원칙적으로 `별칭`을 줄 수 없다 (하이버네이트는 허용하나, 연관 데이터 누락 위험이 있어 지양)
        + 여기서 `별칭`은 `SELECT m.age AS member_age`처럼 값에 이름표를 붙이는 컬럼 별칭이 아니라, `JOIN FETCH t.members m`처럼 페치 대상 엔티티에 이름을 붙여 WHERE 등에서 참조하는 것을 말함
        + 별칭으로 조건을 걸어 컬렉션 일부만 걸러내면, 영속성 컨텍스트에 `일부만 채워진 컬렉션`이 올라감
            ```java
            SELECT t FROM Team t JOIN FETCH t.members m WHERE m.age > 20
            ```
            - team.getMembers() 시 실제 DB엔 20살 이하 멤버도 있지만 `20살 초과 멤버만 담긴 컬렉션`이 반환됨 → 데이터 정합성이 깨져 버그로 이어짐
        + 페치 조인은 연관 데이터를 `온전하게 전부` 가져오는 것이 목적이므로, 컬렉션을 필터링하려면 페치 조인 대신 일반 조인 + 별도 조회를 사용
    - 둘 이상의 `컬렉션`을 페치 조인할 수 없다 (카테시안 곱 발생)
    - 컬렉션을 페치 조인하면 `페이징 API`(setFirstResult, setMaxResults)를 사용할 수 없다
        + 하이버네이트는 경고 로그를 남기고 모든 데이터를 읽어 `메모리에서 페이징` → 데이터 많으면 매우 위험
        + 대안 : `@BatchSize` 또는 `hibernate.default_batch_fetch_size`로 IN 쿼리 최적화

        | 구분 | 일반 조인 | 페치 조인 |
        | --- | --- | --- |
        | 연관 엔티티 조회 | 조인만 하고 SELECT 대상은 아님(프록시/지연로딩) | 연관 엔티티까지 함께 조회 |
        | N+1 문제 | 발생 가능 | 해결 |
        | 별칭 | 가능 | 원칙적으로 불가 |

### 서브 쿼리
+ 쿼리 안에 포함된 또 다른 쿼리
+ sql과 비슷하지만 JPA에서는 몇가지 제약이 있다
    - from 절에서 서브 쿼리 불가
    - select 절은 스펙상 불가능하지만 하이버네이트와 같은 구현체에서는 가능하다
    - where, having에서만 사용가능
    - 엔티티 기준이 아니라 필드(속성) 기준으로 비교해야 할 때 서브 쿼리가 유용
    - where 절
        + 결과 집합에서 `조건을 만족하는 행`만 가져온다
        + =, >, <, between, like, in 등
            ```java
            // 예: 평균 나이 이상인 멤버만 조회
            SELECT m FROM Member m
            WHERE m.age >= (SELECT AVG(m2.age) FROM Member m2)
            ```

    - in 절
        + where 절에서 `특정 값들의 집합에 포함되는지`를 확인
        + where 조건식의 한 부분
            ```java
            // 예: 특정 팀에 소속된 멤버만 조회
            SELECT m FROM Member m
            WHERE m.team.name IN (SELECT t.name FROM Team t WHERE t.name LIKE 'A%')
            ```

    - exist 절
        + 서브쿼리의 조건이 하나라도 만족할 때
            ```java
            // 예: 리뷰가 한 개라도 있는 멤버만 조회
            SELECT m FROM Member m
            WHERE EXISTS (SELECT r FROM Review r WHERE r.member = m)
            ```

### CASE식
    ```java
    SELECT
        CASE
            WHEN m.age >= 60 THEN '노년'
            WHEN m.age >= 30 THEN '중년'
            ELSE '청년'
        END
    FROM Member m
    ```

### COALESCE
+ 여러 값 중에서 Null이 아닌 첫번째 값을 반환
+ 모든 값이 null이면 null 반환
    ```java
    SELECT COALESCE(m.nickName, m.name, '이름없음')
    FROM Member m   
    ```

### NULLIF
+ 두 값이 같으면 null, 다르면 첫번째 값을 반환
    ```java
    SELECT NULLIF(m.name, '관리자')
    FROM Member m
    ```
    - name이 관리자면 null  
    - name이 관리자가 아니면 name 그대로 반환

### 사용자 정의 함수
+ JPQL은 표준함수를 가지고 있고 해당 함수만 지원함
    - length, upper, lower, concat 등
    - 하지만 DB에는 DB 종속적인 자신만의 함수가 있을 수 있다, JPQL에서는 DB 함수는 사용 불가능하지만 방언에 함수를 등록하면 사용이 가능하다

        ```java
        public class MyMySQLDialect extends org.hibernate.dialect.MySQL8Dialect {
            public MyMySQLDialect() {
                super();
                registerFunction("my_custom_func", new StandardSQLFunction("my_custom_func", StandardBasicTypes.STRING));
            }
        }

        @Query("SELECT FUNCTION('my_custom_func', m.name) FROM Member m")
        List<String> callCustomFunc();
        ```
    - 사용하는 JPA 구현체의 방언 클래스를 상속받아 커스텀 함수 등록
    - (참고) 위 `registerFunction` 방식은 Hibernate 5 스타일이며, Hibernate 6(Spring Boot 3)부터는 `FunctionContributor`를 구현하여 등록하는 방식으로 바뀜

    | 방법 | 장점 | 단점 및 주의사항 |
    | --- | --- | --- |
    | 네이티브 쿼리 | 가장 쉽고, 확실함 | 타입 안전성 없음 |
    | Dialect 확장 + JPQL | JPQL에서 바로 사용 가능 | 다이얼렉트, 설정 필요 |
    | Criteria API | 동적 쿼리도 가능 | 코드가 다소 복잡함 |

### 엔티티 직접 사용
+ JPQL에서 엔티티를 직접 사용하면 SQL에서는 해당 엔티티의 `기본 키(PK) 값`으로 변환됨
    ```java
    SELECT COUNT(m) FROM Member m WHERE m = :member
    // 실행 SQL → ... WHERE m.id = ?
    ```
+ 외래 키(연관관계)도 마찬가지로 `외래 키 값`으로 변환됨
    ```java
    SELECT m FROM Member m WHERE m.team = :team
    // 실행 SQL → ... WHERE m.team_id = ?
    ```

### Named 쿼리 (정적 쿼리)
+ 미리 정의하고 이름을 부여해 사용하는 정적 JPQL
+ `애플리케이션 로딩 시점`에 JPQL 문법을 검증하고 미리 파싱 → 문법 오류를 조기에 발견, 성능 이점
    ```java
    @Entity
    @NamedQuery(
        name = "Member.findByName",
        query = "SELECT m FROM Member m WHERE m.name = :name"
    )
    public class Member { ... }

    List<Member> result = em.createNamedQuery("Member.findByName", Member.class)
        .setParameter("name", "강선구")
        .getResultList();
    ```
+ Spring Data JPA의 리포지토리 메서드 위 `@Query`도 로딩 시점에 검증되어 유사한 효과

### 벌크 연산
+ 여러 건을 한 번의 쿼리로 `UPDATE / DELETE`
+ `executeUpdate()` 사용, 영향받은 엔티티 수를 반환
    ```java
    int count = em.createQuery(
        "UPDATE Member m SET m.age = m.age + 1 WHERE m.age >= 20")
        .executeUpdate();
    ```
+ 주의 : 벌크 연산은 `영속성 컨텍스트를 무시`하고 DB에 직접 실행
    - 이미 영속성 컨텍스트에 로딩된 엔티티와 DB 상태가 달라질 수 있음
    - 해결 : 벌크 연산을 `가장 먼저` 실행하거나, 실행 후 `em.clear()`로 영속성 컨텍스트를 초기화한 뒤 다시 조회
