## 1장. JPA 소개

### 패러다임의 불일치
+ 개발은 객체지향적으로 개발하지만 데이터는 관계형 데이터베이스에 저장되면서 발생하는 문제
+ 상속
    - 객체에는 상속이 있지만 RDB에는 상속이 없다
    - Album, Movie, Book이 Item을 상속받는 구조를 관계형으로 저장해야 한다 
        + 관계형 DB의 경우
            - 슈퍼타입/서브타입 테이블을 생성
            - 타입을 분기해서 INSERT 쿼리를 실행

            ```java
            abstract class Item {
                private Long id;
                private String name;
                private int price;
            }

            class Album extends Item { private String artist; }
            class Movie extends Item { private String director; }

            // 이걸 SQL로 저장하려면 타입마다 분기하여 INSERT를 실행해야 한다
            // Album을 저장하려면...
            if (item instanceof Album album) {
                // 1. 부모 격인 ITEM 테이블에 INSERT
                "INSERT INTO ITEM(name, price) VALUES(?, ?)";
                // 2. 자식 ALBUM 테이블에 INSERT
                "INSERT INTO ALBUM(item_id, artist) VALUES(?, ?)";
            }
            // Movie면 또 다른 분기... 조회할 땐 JOIN까지 짜야 한다
            ```
        + JPA의 경우
            - @Inheritance 전략으로 추상화가 가능

            ```java
            @Entity
            @Inheritance(strategy = InheritanceType.JOINED)  // 전략만 고르면 끝
            abstract class Item { ... }

            @Entity
            class Album extends Item { private String artist; }

            em.persist(album);   // JPA가 ITEM + ALBUM 두 테이블에 알아서 나눠 INSERT
            Album find = em.find(Album.class, 1L);  // JOIN도 알아서
            ```

+ 연관관계
    - 객체는 참조(reference)로 관계를 맺는다
        + member.getItem()
    - 테이블은 외래키(foreign key)로 관계를 맺고 JOIN으로 찾는다
    - 객체 참조는 단방향이지만 FK는 양쪽에서 JOIN이 가능하므로 사실상 양방향

    ```java
    class Member {
        private Long id;
        private String name;
        private Team team;   // 참조로 관계를 맺는다
    }

    class Team {
        private Long id;
        private String name;
    }

    member.getTeam().getName();   // 참조를 타고 자연스럽게 이동

    // 이걸 관계형 DB에서 저장하려고 SQL를 짜면..
    // team 객체를 통째로 넣는 게 아니라, FK 값(team_id)을 꺼내서 넣어야 한다
    String sql = "INSERT INTO member(name, team_id) VALUES(?, ?)";
    pstmt.setString(1, member.getName());
    pstmt.setLong(2, member.getTeam().getId());  // 객체 → FK로 강제 변환

    // 객체는 team을 참조하고 싶은데, 테이블은 team_id (PK)밖에 모르므로 변환 코드를 개발자가 직접 작성해야 한다
    // JPA는 해당 작업을 어노테이션 하나로 위임한다

    @Entity
    class Member {
        @Id @GeneratedValue
        private Long id;
        private String name;

        @ManyToOne
        @JoinColumn(name = "team_id")
        private Team team;   // 참조를 그대로 두면 JPA가 FK로 번역해준다
    }

    member.setTeam(team);       // 객체답게 참조를 넣고
    em.persist(member);         // JPA가 알아서 team_id로 INSERT
    Team findTeam = member.getTeam();  // 조회도 객체답게
    ```
    - 방향성 차이
        + 객체는 참조가 단방향이라, 양방향으로 쓰려면 양쪽 클래스에 참조 필드를 각각 만들어야 한다
            - member.getTeam() 을 쓰려면 Member에 team 필드
            - team.getMembers() 를 쓰려면 Team에 members 컬렉션 필드 (총 2개)
        + 테이블은 team_id(FK) 하나로 양쪽에서 JOIN이 가능하므로 애초에 양방향이다
        + 즉, 객체는 필드 2개가 필요하지만 테이블은 FK 1개면 끝 → 이 차이가 패러다임 불일치
    - 근본 원인: **참조는 방향이 있고, FK는 방향이 없다**
        + "객체에는 FK라는 개념이 없어서"는 현상 설명이지 원인이 아니다. 한 단계 아래를 봐야 한다
        + 객체는 참조(주소)로 관계를 맺는데, 참조는 본질적으로 단방향이다
            - `member`가 `team`의 주소를 들고 있다고 해서 `team`이 `member`를 찾을 수 있는 것은 아니다
            - 그래서 양쪽에서 접근하려면 `Team`에도 `members` 필드를 따로 둬야 하고, 이것은 양방향이 아니라 **서로 다른 단방향 2개**다
        + 반면 FK는 **값**이고, JOIN은 어느 쪽에서 걸든 성립한다
            - `member.team_id = team.id`라는 조건 자체에는 방향이 없다 → FK 하나면 처음부터 양방향
        + 이 비대칭(객체 참조 2개 vs FK 1개)이 5장의 **연관관계의 주인**이 필요한 이유로 그대로 이어진다
            - 참조는 둘인데 FK는 하나이므로, 둘 중 어느 쪽을 보고 FK를 쓸지 정해야 한다
+ 객체 그래프 탐색
    - 객체지향에서는 member.getItem().getOrders()처럼 참조를 원하는대로 타고 다닐 수 있다
    - 관계형 DB에서는 처음 조회할 때, JOIN한 범위만 탐색이 가능하고 그 이상은 다시 쿼리를 실행해야 한다

    ```java
    // 이 SQL은 member와 team만 JOIN했다
    String sql = "SELECT m.*, t.* FROM member m JOIN team t ON ...";

    member.getTeam();      // OK, JOIN했으니까 들어있다
    member.getOrder();     // null! order는 SELECT 안 했으니까

    // JPA의 경우
    @ManyToOne(fetch = FetchType.LAZY)
    private Team team;

    Member member = em.find(Member.class, 1L);   // 이 시점엔 team SELECT 안 함
    // ... 
    member.getTeam().getName();   // ← 바로 여기서 team 조회 SQL이 나간다
    ```
    - 근본 문제: 엔티티를 어디까지 신뢰할 수 있는지 알 수 없다
        + 같은 Member 타입이라도 어떤 SQL로 조회했느냐에 따라 내용이 달라진다
            - member+team만 JOIN해서 조회 → member.getOrder()는 null
            - member+order까지 JOIN해서 조회 → member.getOrder()에 값이 있음
        + 핵심은 **null의 의미를 구별할 수 없다**는 것이다
            - `member.getOrder()`가 null일 때, 그것이 "이 회원은 주문이 없다"인지 "주문을 조회하지 않았을 뿐이다"인지 객체만 봐서는 알 수 없다
            - 참고로 order는 member 테이블의 컬럼이 아니라 **다른 테이블(연관 엔티티)**이다. "컬럼이 있냐 없냐"의 문제가 아니라 "어디까지 조회했냐"의 문제다
        + 결국 엔티티 클래스가 아니라 그것을 조회한 SQL(DAO)까지 열어봐야 탐색 가능 범위를 알 수 있다
        + 객체를 자유롭게 신뢰하며 그래프를 탐색할 수 없다는 것이 핵심 문제
    - JPA는 이 문제를 **지연 로딩**으로 푼다
        + null을 남겨두는 대신 프록시를 넣어두고, 실제로 접근하는 순간 SELECT를 실행한다
        + 덕분에 `member.getOrder()`를 항상 신뢰할 수 있다 — 단, **영속성 컨텍스트가 살아있는 동안만**
            - 컨텍스트가 닫힌 뒤 프록시를 건드리면 `LazyInitializationException`이 발생한다
+ 동일성 vs 동등성
    - 객체는 주소값 동등성(==)과 내용 동등성(equals())를 구분한다
    - DB는 PK로 같은 행인지 판단한다
        + SQL로 같은 회원을 조회했을 때, member1 == member2는 false
    - JPA는 같은 트랜잭션 안에서 같은 PK면 같은 인스턴스를 반환하도록 1차 캐시로 동일성을 보장한다

    ```java
    Member m1 = memberDAO.findById(1L);
    Member m2 = memberDAO.findById(1L);

    System.out.println(m1 == m2);   // false!
    // 같은 회원(PK = 1)을 조회했는데, findById가 new Member()로 새 인스턴스를 생성하여 주소값이 다르다
    // DB 입장에서는 같은 행인데, 객체 세계에서는 남남으로 판단
    // ⚠️ "호출한 위치(라인)가 달라서"가 아니다. 같은 라인을 반복 호출해도 매번 new 하므로 결과는 동일하다
    //    원인은 어디까지나 findById 내부에서 매번 새 인스턴스를 만든다는 것

    Member m1 = em.find(Member.class, 1L);   // DB에서 조회 후 1차 캐시에 저장
    Member m2 = em.find(Member.class, 1L);   // 캐시에서 반환 (SELECT 안 나감)

    System.out.println(m1 == m2);   // true! 같은 인스턴스를 보장
    ```
    - JPA가 보장하는 것은 "같은 **값**"이 아니라 "같은 **인스턴스**(동일성)"이다
        + 값이 같은 것(`equals()`)과 주소가 같은 것(`==`)은 다르며, JPA가 보장하는 쪽은 후자다
        + 정확한 표현: **같은 영속성 컨텍스트 안에서, PK를 key로 하는 1차 캐시가 이미 보관 중인 인스턴스를 그대로 반환**한다
        + 따라서 영속성 컨텍스트가 다르면(다른 트랜잭션 등) 이 보장은 성립하지 않는다
